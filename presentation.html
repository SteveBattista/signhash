SignCheckHash

Cryptographically Signed File Integrity Verification

Version 1.0.0 | Manifest Format 0.8.0 | Rust Edition 2024

A high-performance Rust tool for creating and verifying
cryptographically signed file manifests

🤔 The Problem

Traditional Hash Files Are Vulnerable

When you download software or files, providers often give you a list of checksums (hashes). But there's a critical flaw:

  • No guarantee the hash file itself is authentic - An attacker could replace both the files AND the hash list
  • No way to verify integrity - If someone modifies the hash file, you have no way to detect it
  • Trust is manual - You must separately verify the hash file through other channels

Example Attack Scenario:

An attacker compromises a download server, replaces your file with malware, and simply updates the hash list to match. You verify the hashes and think everything is fine! 😱

✅ The Solution: SignCheckHash

Cryptographic Signing Ensures Authenticity

SignCheckHash uses Ed25519 digital signatures to cryptographically sign each file hash in the manifest.

How It Works:

1. Creating a Manifest (sign_hash)

Hash all files in a directory and sign each entry with a private key. The public key is saved separately.

2. Verification (check_hash)

Using the public key, verify that:

  • The manifest hasn't been tampered with
  • Each file hash matches what was originally signed
  • File metadata (size, timestamp) is correct

Key Insight: By not writing out the private key, it becomes impossible to tamper with files and update the manifest later. The public key alone can only verify, not create new signatures.

📝 Using sign_hash

Create Signed Manifests

Basic Usage:

./sign_hash -d ./my_files -o manifest.txt # Hash all files in ./my_files directory # Create signed manifest in manifest.txt # Generate public key (Signpub.txt)

With Options:

./sign_hash -d ./my_files -o manifest.txt -a blake3 -p 8 # Use BLAKE3 hash algorithm # Use 8 threads for processing

With Custom Header:

./sign_hash -d ./my_files -o manifest.txt -i header.txt # Include custom header information in manifest

Available Options:

Flag Description Default
-a, --hash Hash algorithm (SHA256/SHA512/blake3/etc.) SHA256
-s, --signing Signing algorithm ED25519
-u, --public Public key file location Signpub.txt
-o, --output Manifest output file STDOUT
-p, --pool Thread pool size CPU cores
-d, --directory Directory to hash Current dir
-i, --include Header file to include None

✓ Using check_hash

Verify Files and Manifest Integrity

Check Manifest Only:

./check_hash -i manifest.txt -o check_results.txt -m # Verify manifest hasn't been tampered # Check all signatures # Write results to check_results.txt

Check Files Against Manifest:

./check_hash -d ./my_files -i manifest.txt -o check_results.txt # Verify all files match the manifest # Check for tampering, modifications, missing files # Verify manifest signatures

Verbose Mode:

./check_hash -d ./my_files -i manifest.txt -v # Print ALL checks including successful ones # Output to STDOUT

Available Options:

Flag Description Default
-u, --public Public key file location Signpub.txt
-i, --input Manifest file location Manifest.txt
-o, --output Output file location STDOUT
-p, --pool Thread pool size CPU cores
-d, --directory Directory to check Current dir
-v, --verbose Print matches as well as failures False
-m, --manifestonly Check manifest validity only False

🔒 Security Considerations

What Needs Protection?

🔐 Confidentiality (Encryption)

Manifest file should be encrypted if:

  • File names are sensitive
  • File existence is sensitive
  • Hash matching could leak info
  • Rainbow table attacks are a concern

Public key: Does NOT need to be kept secret

✓ Integrity (Tampering)

Manifest & signature must be protected from tampering:

  • Tampered signature = detectable and possible denial of services
  • Tampered manifest = detectable and possible denial of service
  • Both tampered = complete compromise

Protection methods:

  • Multiple storage locations
  • Read signature over phone (64 hex chars)
  • Store in blockchain or immutable DB

🎲 Nonce Security

Each file gets a unique 128-byte random nonce. This makes it cryptographically infeasible (1/2^384 odds) to substitute a line from one manifest to another, even if file names and metadata match.

🔄 Complete Workflow Example

Scenario: Distributing a Software Release

Step 1: Developer Creates Release

./sign_hash -d ./release-v2.0 -o manifest-v2.0.txt -a blake3 # Creates manifest-v2.0.txt (signed hashes) # Creates Signpub.txt (public key) # Private key used and discarded

Developer publishes: release files + manifest-v2.0.txt + Signpub.txt

Step 2: User Downloads Files

User downloads all release files, the manifest, and the public key from the distribution server.

Step 3: User Verifies Manifest Integrity

./check_hash -i manifest-v2.0.txt -m # ✓ Manifest signature valid # ✓ All entries verified # ✓ No tampering detected

Step 4: User Verifies All Files

./check_hash -d ./release-v2.0 -i manifest-v2.0.txt -v # ✓ All files match manifest # ✓ All hashes verified # ✓ All signatures valid # ✓ Files are authentic!

Result: User has cryptographic proof that the files downloaded are exactly what the developer released, with no possibility of tampering.
